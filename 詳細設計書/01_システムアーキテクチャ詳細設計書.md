## 1. システム全体アーキテクチャ

### 1.1 アーキテクチャスタイル

本システムは **シングルページアプリケーション (SPA) + RESTful API** の3層アーキテクチャを採用する。

```
┌──────────────────────────────────────────────────────────────────────┐
│                      プレゼンテーション層                             │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │  React 18 SPA (Vite)                                           │  │
│  │  ├─ Ant Design 5.x UI コンポーネント                             │  │
│  │  ├─ React Router 6 (クライアントサイドルーティング)               │  │
│  │  ├─ Axios HTTP クライアント + インターセプター                   │  │
│  │  ├─ i18next 国際化 (日本語/中国語)                              │  │
│  │  └─ dayjs 日付操作                                              │  │
│  └────────────────────────────────────────────────────────────────┘  │
│                              │ HTTP (JSON / multipart/form-data)      │
│                              ▼                                        │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │                      アプリケーション層                           │  │
│  │  Spring Boot 2.x (組み込みTomcat)                               │  │
│  │  ├─ Spring Security + JWT 認証フィルター                         │  │
│  │  ├─ Controller 層 (REST エンドポイント)                          │  │
│  │  ├─ Service 層 (ビジネスロジック)                                │  │
│  │  ├─ Repository 層 (Spring Data JPA)                             │  │
│  │  └─ Model / DTO 層 (データ変換)                                  │  │
│  └────────────────────────────────────────────────────────────────┘  │
│                              │ JDBC                                   │
│                              ▼                                        │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │                      データ層                                    │  │
│  │  PostgreSQL 16 (データベース) + ローカルファイルシステム (添付)   │  │
│  └────────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────────┘
```

### 1.2 層間通信仕様

| 通信方向 | プロトコル | データ形式 | 認証 |
|----------|-----------|-----------|------|
| フロントエンド → バックエンド | HTTP/1.1 | JSON | JWT (Authorization: Bearer) |
| ファイルアップロード | HTTP/1.1 | multipart/form-data | JWT |
| ファイルダウンロード | HTTP/1.1 | application/octet-stream | JWT |
| バックエンド → データベース | JDBC (TCP 5432) | 独自プロトコル | ユーザー名/パスワード |

### 2.2 コンポーネント設計方針

| 原則 | 説明 |
|------|------|
| 関数コンポーネント | 全コンポーネントを関数コンポーネント + Hooks で実装 |
| 状態管理 | React Context + useState/useReducer (外部ライブラリ不使用) |
| 副作用分離 | データフェッチ・副作用はカスタムフックに集約 |
| UIとロジックの分離 | ページコンポーネントはUIとロジックを分離し、テスト容易性を確保 |
| スタイリング | Ant Design の標準スタイル + 最小限の CSS-in-JS (必要時) |

### 2.3 状態管理設計

```
AuthContext (認証情報)
├── token: string | null          # JWTアクセストークン
├── user: {username, role} | null # ログインユーザー情報
├── login(username, password)     # ログイン処理
├── logout()                      # ログアウト処理
└── isAuthenticated: boolean      # 認証状態判定

各ページコンポーネント内のローカル状態:
├── data[]              # テーブルデータ
├── loading: boolean    # ローディング状態
├── pagination          # ページネーション情報 {page, size, total}
├── filters             # 検索/フィルター条件
├── modalVisible        # モーダル表示状態
└── selectedRecord      # 編集中レコード (null=新規登録)
```

### 2.4 ルーティング設計

| パス | コンポーネント | 認証 | 説明 |
|------|--------------|:--:|------|
| /login | Login.jsx | 不要 | ログイン画面 |
| / | Layout.jsx | 必要 | メインレイアウト（リダイレクト: /employee）|
| /employee | Employee.jsx | 必要 | 社員管理 |
| /customer | Customer.jsx | 必要 | 顧客管理 |
| /attendance | Attendance.jsx | 必要 | 勤務管理 |
| /invoice | Invoice.jsx | 必要 | 請求書・注文書管理 |
| * | (404ページ) | - | 未定義パス |

**ルートガード**: `ProtectedRoute` ラッパーが AuthContext の認証状態をチェックし、未認証時に `/login` へリダイレクトする。

## 4. データフロー詳細

### 4.1 認証フロー

```
[Login.jsx] --POST /api/auth/login {username,password}--> [AuthController]
  → [AuthService] → [UserRepository] → DB検証
  → [JwtTokenProvider] → JWT生成
  → レスポンス {token, type, username, role}
  ← [Login.jsx] localStorage.setItem("token", token) → /employee へ遷移

以降の全リクエスト:
  [api.js Axios Interceptor] → Authorization: Bearer {token} をヘッダー付与
  [JwtAuthFilter] → トークン検証 → SecurityContextHolder に設定
```

### 4.2 CRUD 基本フロー（社員管理を例に）

```
【一覧取得】
[Employee.jsx] useEffect → GET /api/employee?keyword=&page=0&size=10
  → [JwtAuthFilter] → [EmployeeController.getList()]
  → [EmployeeService] → [EmployeeRepository.findAll(Pageable)]
  → Page<Employee> → Page<EmployeeDTO> → ApiResponse
  ← [Employee.jsx] setData() → Ant Design Table 再描画

【新規登録】
[Employee.jsx] モーダル入力 → POST /api/employee {employeeCode, name, ...}
  → [EmployeeController.create()] → @Valid バリデーション
  → [EmployeeService] → [EmployeeRepository.save()]
  → 201 Created → 一覧再取得

【更新】
[Employee.jsx] 編集モーダル → PUT /api/employee/{id} {name, department, ...}
  → [EmployeeController.update()] → 存在チェック → [EmployeeRepository.save()]
  → 200 OK → 一覧再取得

【削除】
[Employee.jsx] 削除確認 → DELETE /api/employee/{id}
  → [EmployeeController.delete()] → 存在チェック → [EmployeeRepository.deleteById()]
  → 200 OK → 一覧再取得
```

### 4.3 ファイル操作フロー

```
【アップロード】
[Employee.jsx] <Upload> → POST /api/employee/{id}/upload (multipart/form-data)
  → [EmployeeController.uploadFile()] → MultipartFile 受信
  → [FileStorageService.store()] → ファイル名UUID化 → ./uploads/employee/{uuid}.ext に保存
  → [EmployeeRepository] → employee.attachmentPath 更新 → save()
  → 200 OK → ファイル名表示

【ダウンロード】
[Employee.jsx] ファイル名クリック → GET /api/employee/{id}/download
  → [EmployeeController.downloadFile()] → [FileStorageService.load()]
  → ResponseEntity<Resource> (Content-Disposition: attachment)
  ← ブラウザダウンロードダイアログ
```

## 6. 非機能要件

### 6.1 パフォーマンス目標

| 指標 | 目標値 |
|------|--------|
| ページ初期表示 | 2秒以内 (First Contentful Paint) |
| APIレスポンス (一覧100件) | 500ms以内 |
| APIレスポンス (登録/更新) | 1秒以内 |
| ファイルアップロード (10MB) | 5秒以内 |
| 同時接続ユーザー数 | 最大50ユーザー |

### 6.2 セキュリティ要件

| 要件 | 実装方式 |
|------|---------|
| 認証 | JWT (HMAC-SHA256) |
| パスワード保管 | BCrypt ハッシュ化 |
| 通信暗号化 | 本番環境では HTTPS (開発環境はHTTP) |
| CORS | 許可オリジン: localhost:3000 (開発) |
| CSRF | APIがステートレスなためJWTで代替対策 |
| SQLインジェクション | Spring Data JPA パラメータバインディング |
| ファイルアップロード制限 | 10MB, 拡張子ホワイトリスト (pdf, doc, docx, xls, xlsx, jpg, png) |
| XSS | Reactのデフォルトエスケープ + 入力サニタイズ |

### 6.3 可用性・保守性

| 項目 | 設計 |
|------|------|
| エラーハンドリング | 全例外は GlobalExceptionHandler で捕捉し、統一レスポンス形式で返却 |
| ログ出力 | SLF4J + Logback、環境別ログレベル設定 |
| ファイルバックアップ | 添付ファイルは日次バックアップ対象 (運用ルール) |
| データベースバックアップ | pg_dump による日次バックアップ (運用ルール) |
